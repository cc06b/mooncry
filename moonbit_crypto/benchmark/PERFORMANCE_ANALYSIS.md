# SHA-256 性能对比分析

## 测试方法

### C 版本优化参数
```bash
gcc -O3 -march=native sha256_c.c -o sha256_c.exe
```

- `-O3`: 最高优化级别
- `-march=native`: 针对本机CPU架构优化
- `clock_gettime()`: 高精度计时

### MoonBit 版本优化
- `const` 预计算常量
- 批量内存操作
- 预分配数组
- 内联关键函数

## 预期性能差距

### 1. 编译器优化差距
| 优化项 | C | MoonBit |
|--------|-----|--------|
| 编译器优化级别 | -O3 | 内置优化 |
| CPU特定优化 | -march=native | 通用 |
| SIMD支持 | SSE/AVX | 暂无 |

### 2. 运行时差距
| 因素 | C | MoonBit |
|------|-----|--------|
| 数组边界检查 | 无 | 可能存在 |
| GC压力 | 无 | 有 |
| 类型转换 | 直接 | 可能需要 |
| 函数调用开销 | 极低 | 较高 |

## 优化方向

### 短期优化（立即可行）
1. **减少函数调用**
   - 完全内联关键路径
   - 批量处理替代逐个调用

2. **优化数据结构**
   - 使用值类型而非引用
   - 预分配避免GC

3. **减少类型转换**
   - 统一使用UInt
   - 避免重复转换

### 中期优化（需要MoonBit更新）
1. **SIMD intrinsics**
   - 一旦MoonBit支持，使用SIMD指令
   - 预期提升 3-5x

2. **更好的编译器优化**
   - 循环展开
   - 寄存器分配优化

### 长期优化（架构改进）
1. **WebAssembly后端优化**
   - 利用WASM的SIMD提案
   - 多线程支持

2. **Native后端**
   - 直接编译为机器码
   - 消除GC开销

## 实际测试步骤


## 关键优化点

### 1. 旋转操作
```c
// C: 单指令
#define ROTR(x, n) (((x) >> (n)) | ((x) << (32 - (n))))
```

```moonbit
// MoonBit: 需要函数调用
fn rotr(x: UInt, n: Int) -> UInt {
  (x >> n) | (x << (32 - n))
}
```

**优化**: 使用inline属性或编译器提示

### 2. 数组访问
```c
// C: 直接指针访问
state[0] += a;

// MoonBit: 可能需要边界检查
state[0] = state[0] + a
```

**优化**: 静态数组 + 编译器优化

### 3. 循环
```c
// C: 完全展开
for (int i = 0; i < 64; i++) { ... }

// MoonBit: 可能需要解释执行
for i in 0..=63 { ... }
```

**优化**: 手动展开关键循环

## 总结

MoonBit 的 SHA-256 实现与 C 的差距主要来自：
1. **运行时开销**: GC、边界检查
2. **编译器优化**: 无法利用CPU特定特性
3. **SIMD支持**: 尚未支持SIMD指令集

**建议**: 对于性能敏感的应用，可以考虑：
- MoonBit 用于开发阶段快速迭代
- 关键路径使用 C/Rust 实现
- MoonBit 专注于业务逻辑和高层抽象
